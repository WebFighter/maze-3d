架构方案选择

Three.js 与物理引擎的集成方式，这是整个项目最核心的决策

A

React Three Fiber + use-cannon

React 生态的 3D 框架。用 JSX 声明式写 Three.js 场景，use-cannon 基于 Cannon-es 提供物理。代码风格统一（全是 React 组件），生态丰富（后处理、控制器等都有现成库），社区活跃。但 React 组件模式与游戏循环有一定摩擦，且 use-cannon 是轻量物理引擎不是 Ammo.js。

B

React Three Fiber + Ammo.js 自行集成（推荐）

用 React Three Fiber 渲染（声明式写法、React 状态管理），但物理层直接用 Ammo.js 通过 useFrame 游戏循环手动同步。兼具 R3F 的开发效率和 Ammo.js 的完整物理能力。物理同步逻辑需要自己写，但完全可控。

C

原生 Three.js + Ammo.js

不用 React Three Fiber，直接在 React 组件中用原生 Three.js + Ammo.js。完全控制渲染循环和物理步进，无抽象层开销。但需要手动管理 Three.js 对象生命周期，与 React 状态管理脱节，代码量大。

对比总结

维度 A. R3F + use-cannon B. R3F + Ammo.js C. 原生 Three.js
物理能力 轻量 (Cannon-es) 完整 (Ammo.js/Bullet) 完整 (Ammo.js/Bullet)
开发效率 最高 高 较低
可控性 受限于库封装 完全可控 完全可控
游戏机制扩展 传送门/跳台难实现 斜坡/跳台/传送门都行 斜坡/跳台/传送门都行
代码量 中等 中等 大
